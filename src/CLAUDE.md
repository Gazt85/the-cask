# The Cask вЂ” project context for Claude Code

## What is this?

A whisky collection management web app targeting enthusiasts in Uruguay and Argentina.
Side project built by a solo developer. MVP first, scale later.

The name "The Cask" references the small casks used in whisky maturation.

---

## Why this market

- Uruguay is the **2nd highest per-capita whisky consumer in the world** (only behind France), at ~1.77L/person/year
- Argentina is the lowest whisky consumer in South America but a growing market
- No whisky app exists in Spanish with meaningful local coverage
- Global competitors (Distiller, WhiskyBase) have no Spanish localization and poor regional data
- The app fills the same niche Vivino fills for wine, but for whisky in the RГ­o de la Plata region

---

## Tech stack

### MVP (current)
- **Frontend + API**: Next.js 14+ with App Router, TypeScript, Tailwind CSS, shadcn/ui
- **Backend as a service**: Supabase (PostgreSQL + Auth + Storage)
- **Deployment**: Vercel (frontend), Supabase cloud

### Post-MVP (when it scales)
- Replace Supabase API layer with a custom **ASP.NET Core (.NET 8)** backend
- Keep **PostgreSQL** (do NOT migrate to MySQL вЂ” Supabase already uses Postgres)
- Keep Next.js frontend unchanged, just reroute API calls
- Developer has strong C# experience, zero Node.js experience вЂ” this is why .NET was chosen over Node

---

## MVP scope (what we are building now)

Only these features ship in v1:

1. **Virtual whisky cabinet** вЂ” add bottles, track quantity, set status (sealed / open / finished)
2. **Whisky catalogue** вЂ” shared DB of whiskies; users search and pick from it when adding a bottle
3. **Personal tasting notes** вЂ” nose, palate, finish, score (1вЂ“100) per bottle; multiple notes over time
4. **Personal rating** вЂ” overall 1вЂ“100 score per bottle in collection

Explicitly **out of scope for MVP**:
- Label scanning (post-MVP вЂ” requires Vision API integration)
- Social feed / following users
- Whisky events nearby
- In-app purchases / marketplace
- Public profiles

## Database schema

Four tables in PostgreSQL (Supabase):

```
profiles            вЂ” linked 1:1 to auth.users via auth_user_id
whiskies            вЂ” shared catalogue, seeded from dataset, read-only for users
collection_bottles  вЂ” user's personal inventory (FK to profiles + whiskies)
tasting_notes       вЂ” multiple tasting notes per bottle (FK to collection_bottles)
```

### Key design decisions
- `profiles` is separate from `auth.users` вЂ” never touch Supabase's auth schema directly
- `whiskies` is a shared catalogue вЂ” if a whisky exists, all users reference the same row
- `bottle_status` is a PostgreSQL enum: `'sealed' | 'open' | 'finished'`
- `rating` on `collection_bottles` = overall personal score (1вЂ“100)
- `score` on `tasting_notes` = score for that specific tasting session (1вЂ“100)
- Full-text search index on `whiskies` (name + distillery) via `to_tsvector`
- A trigger `handle_new_user` auto-creates a `profiles` row on signup

### RLS policies summary
- `profiles`: user can read/update only their own row
- `whiskies`: public read, no user writes (admin/seed only)
- `collection_bottles`: full CRUD, scoped to own `profile_id`
- `tasting_notes`: full CRUD, scoped to own `profile_id`

## Supabase client pattern

Two clients вЂ” always use the right one:

```ts
// In client components ('use client')
import { createClient } from '@/lib/supabase'
const supabase = createClient()

// In server components, server actions, route handlers
import { createServerSupabaseClient, getProfileId } from '@/lib/supabase'
const supabase = await createServerSupabaseClient()
```

`getProfileId()` is a helper that resolves `auth.uid()` в†’ `profiles.id` in one call.
Use it at the top of every server action that writes data.

## Conventions

- All DB columns: `snake_case`
- All TypeScript types: `PascalCase`
- All files and folders: `kebab-case`
- Server actions live in `lib/actions/` вЂ” one file per domain
- No ORM вЂ” raw Supabase query builder only for MVP
- Prefer server actions over API route handlers for mutations
- Dates stored as `timestamptz`, always UTC, format on the client
